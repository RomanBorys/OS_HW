Домашнє завдання №7 — Аналіз життєвого циклу контейнера

1. Запуск контейнера

Контейнер було запущено з використанням образу Python та простого HTTP-сервера.

sudo docker run -d -p 8080:8000 python:3.12-slim python -m http.server 8000

Контейнер працює у фоновому режимі та пробросив порт 8080 на порт 8000 всередині контейнера.


2. Перевірка запущеного контейнера

sudo docker ps

Результат:

Container ID: `200b357b40c6`
Image: `python:3.12-slim`
Status: `Up`
Port mapping: `8080 -> 8000`

3. Процес PID 1 у контейнері

Для перевірки процесів усередині контейнера використано:

sudo docker top 200b357b40c6

Результат:

Головним процесом (PID 1) є:

python -m http.server`

Пояснення

PID 1 у контейнері — це головний процес, з якого починається виконання контейнера. У Docker життєвий цикл контейнера напряму залежить від цього процесу: якщо він завершується — контейнер зупиняється.

4. Перегляд логів контейнера

sudo docker logs 200b357b40c6

Пояснення

Логи контейнера формуються зі стандартного виводу (stdout) та стандартного потоку помилок (stderr) процесу, який виконується всередині контейнера. У даному випадку це HTTP-сервер Python.


5. Зупинка контейнера

sudo docker stop 200b357b40c6

Пояснення

При зупинці контейнера Docker надсилає сигнал `SIGTERM` процесу PID 1. Якщо процес не завершується коректно — надсилається `SIGKILL`.


6. Перевірка зупинки

sudo docker ps

Контейнер більше не відображається у списку активних, що підтверджує його успішне завершення.

Висновок
У ході роботи було:

запущено Docker-контейнер із веб-сервером;
досліджено процес PID 1 усередині контейнера;
проаналізовано механізм життєвого циклу контейнера;
отримано та інтерпретовано логи контейнера;
виконано коректне завершення роботи контейнера.

Було встановлено, що в Docker життєвий цикл контейнера повністю залежить від головного процесу (PID 1), а всі логи отримуються через стандартні потоки виводу процесу.
